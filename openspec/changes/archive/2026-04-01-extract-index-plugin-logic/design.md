## Context

当前插件入口文件直接声明 `resolvedConfig`、`warn`，并在 `closeBundle` 中串联配置加载和预渲染执行。虽然逻辑规模不大，但入口文件已经同时承载公开入口和内部运行时细节。

仓库现有结构已经将配置解析、路径发现、预渲染等逻辑分别放在独立模块中，因此将插件编排继续拆分为独立内部模块，符合当前代码组织方式，也能让入口保持一致的“薄导出”风格。

## Goals / Non-Goals

**Goals:**

- 让 `src/index.ts` 只负责默认导出插件工厂。
- 将插件内部运行时编排抽离到独立模块，并复用现有 `load-config` 与 `prerender` 模块。
- 保持运行时行为、告警语义和测试结果不变。

**Non-Goals:**

- 不修改 `react-ssg.config.ts` 的配置格式。
- 不改变预渲染输出目录、路径发现策略或失败回退策略。
- 不引入新的公开导出或新的外部依赖。

## Decisions

1. 新增内部模块承载插件实现。
原因：入口文件的主要问题是职责混杂，而不是单个函数过长。把完整插件实现迁移到独立模块，能最直接地让入口恢复为薄封装。
备选方案：仅抽出 `closeBundle` 处理函数。该方案仍需在入口文件中保留运行时状态与告警辅助函数，入口依然不够干净，因此不采用。

2. 保留现有警告格式和 `closeBundle` 触发时机。
原因：本次是纯重构，目标是改善结构而非改变行为。继续复用现有 `loadReactSsgConfig` 与 `prerenderBuild` 调用链，可以把风险控制在模块边界重组。
备选方案：顺带统一日志或改名。该方案会扩大改动面，增加不必要的行为回归风险，因此不采用。

3. 通过集成测试验证对外行为不变。
原因：现有测试已经覆盖缺失配置、非法配置、路由预渲染和页面级失败回退，适合作为重构回归保障。额外补充入口瘦身后的最小断言即可。

## Risks / Trade-offs

- [风险] 逻辑迁移时漏掉 `ResolvedConfig` 缓存或告警分支，导致构建行为变化。
  → 缓解：保持原有控制流顺序不变，并通过现有 `vitest` 集成测试回归验证。
- [权衡] 新增一个内部文件会带来轻微模块跳转成本。
  → 缓解：入口职责更清晰，长期维护收益高于单文件查看的便利性损失。
