## Context

当前仓库是一个通用 React 组件模板，尚未具备 Vite 插件的能力边界、配置加载流程和构建期预渲染链路。本次变更需要把仓库转向一个“构建期为传统 React SPA 生成静态 HTML”的插件，同时保持与普通 Vite 构建流程兼容：在没有配置、配置非法或页面渲染失败时，不能阻断原有 CSR 产物。

约束条件如下：

- 所有配置和文档产物使用简体中文。
- `v1` 只支持 `routes` 或 `app` 两种输入，不支持 `router`。
- `v1` 不处理 React Router `loader`，不引入浏览器 API polyfill。
- 构建失败回退需要细分为“全局跳过预渲染”和“单页面回退 CSR”两类。
- 测试必须遵循 TDD，使用 Vitest 覆盖配置解析、路径发现、输出生成和失败容错。

## Goals / Non-Goals

**Goals:**

- 提供一个可独立加载的 `react-ssg.config.ts` 配置入口，并通过 `defineReactSsgConfig` 提供类型友好 API。
- 支持两种预渲染模式：
  - 路由模式：`history + routes + paths`
  - 单页模式：`app`
- 在 `browser` history 下自动发现静态路由，并允许通过 `paths` 补充动态路径实例。
- 在 `hash` history 下仅预渲染默认首屏 `index.html`。
- 在单页面或单路由预渲染失败时仅跳过对应目标，不中断其它页面，也不阻断原始 CSR 构建。

**Non-Goals:**

- 不支持直接传入 `router`、`createBrowserRouter()` 或 `createHashRouter()` 实例。
- 不在 `v1` 中运行 `loader`、注入 hydration data 或实现真正的 SSG 数据预取。
- 不在 `v1` 中提供 `jsdom`、`localStorage` 等 polyfill 能力。
- 不尝试兼容非 React Router 的多页面路由系统或自动扫描整个项目入口。

## Decisions

### 1. 使用独立配置文件 `react-ssg.config.ts`

决策：插件运行时强约定读取项目根目录下的 `react-ssg.config.ts`，并通过 `defineReactSsgConfig` 统一导出对象或函数配置。

原因：

- 将“预渲染规则”与 `vite.config.ts` 的构建配置解耦，便于扩展和文档表达。
- 便于把 `routes`/`app`、`history`、`paths` 等领域配置放入单独类型系统。
- 配置缺失时可以明确降级为普通 CSR 构建，而不需要污染用户的 Vite 配置。

备选方案：

- 直接在 `vite.config.ts` 传插件选项：实现简单，但会把预渲染领域配置与 Vite 构建配置混在一起，后续扩展成本更高。
- 通过字符串模块路径引用 `routes`：实现可控，但用户体验较差，且路径重构较脆弱。

### 2. 使用互斥配置模型区分“路由模式”和“单页模式”

决策：配置类型设计为互斥联合，要求用户在 `routes` 与 `app` 中二选一。

原因：

- 两类输入的输出语义不同：路由模式需要路径发现与多目标渲染，单页模式只需要渲染根入口。
- 互斥类型能在编译期阻止不合法组合，减少运行时歧义。

备选方案：

- 用单一宽泛配置对象同时接受多个字段：灵活但容易产生冲突，例如同时传 `routes` 和 `app`。

### 3. 在 `browser` history 下做静态路径发现，在 `hash` history 下只渲染默认首屏

决策：

- `browser` history：从 `RouteObject[]` 中提取静态路径，并合并用户提供的 `paths`。
- `hash` history：不尝试为 `#/foo` 等 hash 片段生成独立 HTML，仅渲染站点默认首屏。

原因：

- `browser` 路由的 URL 会参与静态托管路径映射，适合输出多份 HTML。
- `hash` 片段不会参与服务器请求，多份 HTML 无法和 hash 片段建立一一映射。

备选方案：

- 为 hash 中的每个路径生成独立产物：静态托管端无法基于 hash 选择不同 HTML，没有实际收益。

### 4. 预渲染失败按“配置级”与“页面级”分层回退

决策：

- 缺失配置文件或配置非法：整体跳过预渲染，保留普通 CSR 构建输出。
- 单个页面渲染失败：仅跳过该页面的预渲染，继续处理其它页面。

原因：

- 与“不要阻断原有 Vite 构建”的产品目标一致。
- 页面级容错能够提高整体命中率，避免一个页面异常导致全部预渲染失效。

备选方案：

- 任一页面失败则全站放弃预渲染：实现简单，但对真实项目的容错能力较差。

### 5. `v1` 仅渲染 HTML，不引入数据加载与浏览器环境模拟

决策：`v1` 使用服务端渲染 API 直接生成 HTML，不执行 React Router `loader`，也不提供 polyfill。

原因：

- 先聚焦“传统 SPA 构建期首屏 HTML”这一最小闭环。
- 避免 hydration data、请求上下文、浏览器 API 兼容性等问题同时进入首版。

备选方案：

- 同时引入 `loader` 执行：更接近完整 SSG，但显著提升复杂度。
- 默认使用 `jsdom`：可扩大兼容面，但会引入额外依赖、运行时开销和更多 hydration 偏差风险。

## Risks / Trade-offs

- [用户在渲染阶段直接访问浏览器 API] → 不做 polyfill 时，页面会触发该路由预渲染失败；通过“按页面回退 CSR”减轻影响，并在文档中明确约束。
- [配置文件直接导入应用模块带来副作用] → 需要限制配置加载与渲染流程的执行边界，并在实现中区分“配置校验失败”和“页面渲染失败”。
- [静态路径发现无法覆盖动态路由参数] → 通过 `paths` 显式补充，并在 spec 与文档中明确这是必需配置。
- [从组件模板迁移为插件仓库会影响现有示例和测试结构] → 在任务中显式包含模板清理、playground 调整和 README 重写，保证对外表现一致。
