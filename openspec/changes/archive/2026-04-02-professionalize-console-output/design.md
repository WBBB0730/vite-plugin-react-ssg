## Context

当前插件的输出逻辑主要由 `src/react-ssg-plugin.ts` 直接调用 `console.warn` 完成，而具体错误消息又分散在 `src/load-config.ts` 与 `src/prerender.tsx` 中。这样的结构带来两个问题：一是输出语言和语气不统一，二是调用点分散，不利于后续继续打磨日志体验。

用户本次明确要求将所有控制台输出改为英文，并参考 `next build` 的输出风格。进一步讨论后，输出目标已经收敛为两层：默认模式保留克制的阶段日志，详细模式仅在完成摘要后追加逐路由结果列表。结合当前项目规模，这更适合通过“统一 logger + 可配置输出粒度”的方式实现，而不是在各处直接拼接字符串。

## Goals / Non-Goals

**Goals:**

- 将插件相关控制台输出统一为英文。
- 为构建期预渲染流程提供统一的日志输出入口。
- 通过 `logLevel` 支持 `silent`、`normal`、`verbose` 三档输出粒度，并保证默认行为简洁稳定。
- 输出具备明确的标题、留白、阶段提示、警告和单行完成摘要，整体风格更接近 `next build`。
- 保持当前预渲染行为、回退语义和文件产物不变。

**Non-Goals:**

- 不新增任何外部日志依赖。
- 不引入耗时统计、体积分析、表格报告或复杂交互式输出。
- 不暴露自定义 formatter 或自定义符号映射能力。

## Decisions

1. 对外使用 `logLevel`，对内统一命名为 `logger`。
原因：`logLevel` 是用户可见配置项，语义是“输出多少”；`logger` 是内部实现对象，语义是“如何输出”。这组命名天然成对，比 `logLevel + reporter` 更一致。
备选方案：继续沿用 `reporter`。该方案并非错误，但会让用户配置与内部实现的术语不够统一，因此不采用。

2. `logLevel` 采用 `silent | normal | verbose` 三档，默认值为 `normal`。
原因：当前真正需要控制的是“构建输出粒度”，而不是传统后端日志系统里的 debug/info/warn/error 等通用级别。三档已经足够覆盖静默、默认和详细排障场景。
备选方案：设计成 `debug/info/warn/error`。该方案更像通用 logger 配置，但对当前插件的输出类型并不贴切，因此不采用。

3. `prerenderBuild` 返回结构化 summary 与逐路由结果，交由 logger 按 `logLevel` 决定展示粒度。
原因：如果要同时支持单行完成摘要和 `verbose` 结果列表，插件入口需要拿到总目标数、成功数、跳过数，以及每个目标路径的最终状态。由预渲染模块返回结构化结果，最清晰也最便于测试。
备选方案：让 logger 在执行过程中自行累计所有状态。该方案会把输出和执行状态耦合在一起，不如由 `prerenderBuild` 直接返回结果清晰，因此不采用。

4. `normal` 与 `verbose` 共享完全一致的核心输出，`verbose` 仅在完成摘要后追加结果列表。
原因：详细模式应该是默认模式的超集，而不是另一套表达体系。这样既能保证阅读体验稳定，也能减少测试和维护成本。
备选方案：让 `verbose` 改写 warning 或 summary 的表述方式。该方案会造成不同模式间语义不一致，因此不采用。

5. 完成信息与 summary 合并为单行 completed 摘要，并在各块之间保留空行。
原因：单行摘要可以避免“完成提示 + summary”之间的重复，同时保留阶段完成感；适度留白能让日志分组更接近 `next build` 的阅读体验。
备选方案：保留独立成功行与 summary 行。该方案信息有重叠，显得啰嗦，因此不采用。

## Risks / Trade-offs

- [风险] 修改 `prerenderBuild` 返回值后，可能影响现有调用链。
  → 缓解：仅调整内部调用接口，并通过现有集成测试覆盖成功、配置失败、页面级失败以及不同 `logLevel` 场景。
- [风险] 日志断言过于依赖完整字符串，后续微调文案会造成测试脆弱。
  → 缓解：测试聚焦关键阶段和核心句式，避免对无关细节做过度绑定。
- [权衡] 增加 `logLevel` 与逐路由结果列表后，内部状态会比当前略复杂。
  → 缓解：保持公开配置最小化，且让 `verbose` 只在 completed 摘要后追加结果块，不扩散到其它输出分支。
